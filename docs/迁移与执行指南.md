# Muse DataLoad PostgreSQL 迁移与执行指南

Last updated: 2026-04-27 16:55 Asia/Shanghai

本文用于按“近两年全站元数据 JSONL -> PostgreSQL -> 官方角色榜单 -> 榜单快照 -> 定向下载任务”的路线执行。全站阶段不要下载图片。

## 0. 执行原则

- PostgreSQL 是唯一正式数据库。
- JSONL 是原始真源，PostgreSQL 是分析、聚合、榜单和任务状态主库。
- 图片下载不属于全站同步流程，只能由榜单角色或榜单快照驱动。
- 主跑按 monthly 分片，失败后只补失败片。
- manifest 必须固化到 `sync_job / sync_shard / job_log`，避免只依赖文件状态。

## 14. 故障处理

- 429 或 5xx：降低 `CRAWLER_RPS`，然后重跑同一条 `--resume` 命令。
- 单个分片异常：优先用 `scripts/retry_failed_shards.py` 按 manifest 补跑。
- JSONL 有重复 ID：manifest 会记录 `duplicate_ids`；PostgreSQL 导入按 post id upsert。
- 磁盘不足：先保留 JSONL，延后导入 PostgreSQL；必要时使用季度分片或外接硬盘作为 `--output-root`。
- Docker API 返回 `/app/output/...`：这是容器内路径，宿主机对应 `output/...`。
- 代理中断：先恢复代理，再执行失败片补跑；不要直接整轮重跑。

## 15. 每周增量榜单更新

`scripts/weekly_update_rankings.py` 是调度入口，默认只同步 Danbooru 元数据并刷新榜单，不下载图片。它会按窗口执行：

1. 生成本次日期窗口和 manifest 路径。
2. 调用 `sync_recent_posts.py` 生成增量 JSONL。
3. 调用 `import_jsonl_fast.py` 批量导入 JSONL。
4. 调用 `sync_manifest_to_db.py` 固化分片状态。
5. 对受影响月份执行增量月度聚合。
6. 构建综合榜和新兴榜。
7. 执行 JSON/CSV 校验。
8. 写入 `sync_job`、`job_log`，并保留新的 `ranking_snapshot`。

### 15.0 性能基线采集

```powershell
.\.venv\Scripts\python.exe scripts\collect_performance_baseline.py `
  --database-url $env:DATABASE_URL `
  --explain `
  --output output\metadata\performance_baseline.json
```

该命令只读数据库，用于记录核心表行数、post 时间范围、最近榜单快照和关键 SQL 查询计划。正式性能验收时，应在执行导入、聚合、榜单构建命令前后各采集一次，并把耗时和报告路径写入 `docs/验收报告.md`。

### 15.1 Dry-run

```powershell
.\.venv\Scripts\python.exe scripts\weekly_update_rankings.py `
  --database-url $env:DATABASE_URL `
  --dry-run
```

dry-run 只输出计划窗口、manifest 路径和步骤，不创建榜单快照。

### 15.2 手动执行指定窗口

```powershell
.\.venv\Scripts\python.exe scripts\weekly_update_rankings.py `
  --database-url $env:DATABASE_URL `
  --start-date 2026-05-08 `
  --end-date 2026-05-15 `
  --resume `
  --output-root ./output
```

联网冒烟或验收时可以加 `--sync-limit 200` 限制抓取条数，确认链路后再去掉限制。

如果不传 `--start-date`，脚本会查找 `weekly-ranking-update:*` 中最近一次成功任务的 `window_end`，从下一天开始；如果没有历史成功任务，则使用 `--lookback-days`，默认 7 天。

### 15.3 Windows Task Scheduler 示例

```powershell
$action = New-ScheduledTaskAction `

  -Argument "scripts\weekly_update_rankings.py --database-url $env:DATABASE_URL --resume" `

$trigger = New-ScheduledTaskTrigger -Weekly -DaysOfWeek Monday -At 03:00

Register-ScheduledTask `

  -Description "Weekly Danbooru metadata sync and ranking refresh"
```

### 15.5 查看最近任务和榜单

```sql
select job_key, status, params->>'window_start' as window_start, params->>'window_end' as window_end, completed_at
from sync_job
where job_key like 'weekly-ranking-update:%'
order by completed_at desc nulls last
limit 10;

select id, ranking_type, generated_at, export_json_path, export_csv_path
from ranking_snapshot
order by generated_at desc
limit 10;
```

### 15.6 失败恢复

- 如果同步失败：修复网络或限速配置后，用同一 `--start-date` / `--end-date` 加 `--resume` 重跑。
- 如果导入失败：确认 JSONL 文件存在后重跑同一窗口；导入按 post id upsert，重复运行不会重复写入 post/post_tag。
- 如果榜单构建失败：上一期 `ranking_snapshot` 不会被删除，可继续作为线上可用榜单。
- 如果需要补跑历史窗口：显式传入 `--start-date` 和 `--end-date`，脚本会生成独立 job key，例如 `weekly-ranking-update:2026-05-08:2026-05-15`。
