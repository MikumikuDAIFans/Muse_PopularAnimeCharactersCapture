# Muse DataLoad 迁移与执行指南

Last updated: 2026-04-25 18:10 Asia/Shanghai

本文用于把项目迁移到新笔记本后，按“近两年全站元数据 JSONL -> 角色候选榜 -> 榜单角色定向样本下载”的路线执行。全站阶段不要下载图片。

## 0. 稳定执行策略

建议把长跑拆成四段，而不是一次性把 24 个月赌在一个长会话里：

1. 先做小流量灰度，确认代理、权限、manifest 和 JSONL 输出正常。
2. 再做主跑，按 monthly 分片收集两年元数据。
3. 主跑结束后只补跑失败片，不重抓已经成功的大月。
4. 全部分片稳定后，再做候选榜聚合和后续定向样本导出。

保底原则：

- 代理和同步进程分离，代理单独守护，不与单次 SSH 会话同生共死。
- 主跑时使用 `monthly` 分片，降低单片失败的爆炸半径。
- 失败片补跑优先基于 manifest 判断“缺 JSONL 或 lines=0”，避免把已成功的百万级数据重抓一遍。
- `worker.errors > 0` 但 `lines > 0` 的 shard 先保留，只有业务上需要极致完整时再二次补跑。
- 每轮重跑都产出独立 retry manifest，避免混淆原始主跑记录。

## 1. 新笔记本准备

建议预留：

- 源码与虚拟环境：5GB。
- 近两年 JSONL、manifest、临时日志：至少 100GB。
- 如果后续把全部 JSONL 导入 SQLite 并展开 `post_tag`：额外预留 100GB 以上，具体以实测为准。

安装工具：

- Git
- Python 3.12 或更新版本
- Node.js LTS
- Docker Desktop

克隆仓库：

```powershell
git clone https://github.com/MikumikuDAIFans/Muse_PopularAnimeCharactersCapture.git Muse_DataLoad
cd Muse_DataLoad
```

创建 Python 环境：

```powershell
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install --upgrade pip
.\.venv\Scripts\pip.exe install -r backend\requirements.txt
```

可选 `.env`：

```powershell
@"
DANBOORU_LOGIN=
DANBOORU_API_KEY=
CRAWLER_RPS=1.0
OUTPUT_ROOT=./output
DATABASE_URL=sqlite+aiosqlite:///./muse_dataload.db
"@ | Set-Content -Encoding UTF8 .env
```

没有 Danbooru API key 也可以跑公开元数据，但长跑更建议配置账号与 key，并保持低 RPS。

## 3. 灰度测试元数据同步

先跑每月 100 条，确认网络、权限、输出路径和 manifest 都正常：

```powershell
.\.venv\Scripts\python.exe scripts\sync_recent_posts.py `
  --recent-months 24 `
  --shard monthly `
  --resume `
  --skip-import `
  --task-id-prefix 924000 `
  --limit 100
```

检查输出：

```powershell
Get-ChildItem output\metadata
Get-Content output\metadata\sync_recent_posts_manifest.json -Raw
```

灰度输出正常后，可以删除灰度 JSONL 或换一个新的 `--task-id-prefix` 跑正式任务。

## 4. 正式同步近两年全站元数据

推荐 monthly 分片，便于恢复、核对和局部重跑：

```powershell
.\.venv\Scripts\python.exe scripts\sync_recent_posts.py `
  --recent-months 24 `
  --shard monthly `
  --resume `
  --skip-import `
  --task-id-prefix 924000
```

中断后直接重跑同一条命令。脚本会读取：

```text
output/metadata/task_<task_id>.checkpoint.json
output/metadata/task_<task_id>_posts.jsonl
```

并从 Danbooru 搜索游标 `last_cursor` 继续。不要改 `--task-id-prefix`，否则会生成新的分片文件。

如果希望减少文件数量，可使用季度分片：

```powershell
.\.venv\Scripts\python.exe scripts\sync_recent_posts.py `
  --recent-months 24 `
  --shard quarterly `
  --resume `
  --skip-import `
  --task-id-prefix 924000
```

推荐正式跑法：

```powershell
.\.venv\Scripts\python.exe scripts\sync_recent_posts.py `
  --recent-months 24 `
  --shard monthly `
  --resume `
  --skip-import `
  --task-id-prefix 924500
```

建议不要在第一次正式跑时混用 `--limit`。`--limit` 只用于灰度和网络测算。

### 4.1 失败片补跑

如果主跑结束后 manifest 中有 shard 缺少 JSONL 或 `lines=0`，直接按 manifest 补跑失败片：

```powershell
.\.venv\Scripts\python.exe scripts\retry_failed_shards.py `
  --manifest output\metadata\sync_recent_posts_manifest.json `
  --skip-import
```

如果只想补跑指定 task id：

```powershell
.\.venv\Scripts\python.exe scripts\retry_failed_shards.py `
  --manifest output\metadata\sync_recent_posts_manifest.json `
  --task-ids 924505 924506 924509 `
  --skip-import
```

如果需要把 `errors > 0` 但已有行数的 shard 一并重跑：

```powershell
.\.venv\Scripts\python.exe scripts\retry_failed_shards.py `
  --manifest output\metadata\sync_recent_posts_manifest.json `
  --retry-nonzero-errors `
  --skip-import
```

补跑产物：

```text
output/metadata/sync_recent_posts_retry_manifest.json
```

### 4.2 何时算通过

主跑或补跑后，至少满足以下条件再进入候选榜聚合：

- `sync_recent_posts_manifest.json` 或 retry manifest 中，目标 shard 都存在 JSONL 文件。
- 每个目标 shard 的 `jsonl.lines > 0`。
- `duplicate_ids` 和 `invalid_lines` 没有异常飙升。
- 失败片只剩少量、且业务上确认可接受；否则继续补跑。

## 5. 生成 JSONL 角色候选榜

全站 JSONL 同步完成后，先不导入 SQLite，直接流式聚合：

```powershell
.\.venv\Scripts\python.exe scripts\build_character_candidates_from_jsonl.py `
  --recent-months 24 `
  --top-n 500 `
  --min-count 50
```

输出：

```text
output/exports/character_candidates_recent_24m_top_500.json
output/exports/character_candidates_recent_24m_top_500.csv
```

这份候选榜用于人工抽检、调整阈值和决定后续定向样本下载范围。

## 6. 可选：导入 SQLite 并生成正式榜单

如果新笔记本磁盘和时间允许，再把 JSONL 分片导入 SQLite：

```powershell
Get-ChildItem output\metadata\task_*_posts.jsonl | ForEach-Object {
  .\.venv\Scripts\python.exe scripts\import_jsonl_fast.py $_.FullName --db muse_dataload.db --batch-size 5000
}
```

导入后生成正式综合榜：

```powershell
.\.venv\Scripts\python.exe scripts\build_character_list.py --recent-months 24 --top-n 500 --min-count 50
```

注意：完整导入 `post_tag` 会显著膨胀 SQLite。若只是为了筛选角色，优先使用第 5 步的 JSONL 候选榜。

## 7. 只对榜单角色定向样本下载

启动 Docker 后端后，通过 API 对选定角色下载小样本：

```powershell
$body = @{
  character_tag = "hatsune_miku"
  limit = 20
  download_images = $true
  clean_target_dir = $true
} | ConvertTo-Json

Invoke-RestMethod -Method Post `
  -Uri "http://127.0.0.1:8001/api/datasets/export" `
  -ContentType "application/json" `
  -Body $body
```

校验：

```powershell
.\.venv\Scripts\python.exe scripts\validate_dataset.py output\dataset\hatsune_miku
```

定向下载原则：

- 只下载进入候选榜或正式榜单的角色。
- 先每角色 5 到 20 张抽检，再扩大。
- 不执行全站图片下载。

## 9. 故障处理

常见处理方式：

- 429 或 5xx：降低 `.env` 中 `CRAWLER_RPS`，然后重跑同一条 `--resume` 命令。
- 单个分片异常：优先用 `scripts/retry_failed_shards.py` 做按 manifest 补跑；只有在某个 shard 文件明显损坏时，才删除对应 `task_<id>_posts.jsonl`、`task_<id>.checkpoint.json` 和 `task_<id>_posts.invalid.jsonl` 后重跑。
- JSONL 有重复 ID：manifest 会记录 `duplicate_ids`；导入脚本按 post id upsert，少量重复不影响入库。
- 磁盘不足：先保留 JSONL，不导入 SQLite；必要时改用季度分片或外接硬盘作为 `--output-root`。
- Docker API 返回 `/app/output/...`：这是容器内路径，宿主机对应 `output/...`。
- 代理中断：先恢复代理，再执行失败片补跑；不要直接整轮重跑。
- 某轮主跑中途失败：保留已有 `task_<id>_posts.jsonl` 和 checkpoint，重新执行同一条命令；若该轮已经结束且部分月份失败，再使用 retry 脚本做补跑。
