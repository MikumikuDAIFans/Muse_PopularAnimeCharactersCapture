## 目标
在保留现有 PostgreSQL-only 架构和 Danbooru 元数据优先策略的前提下，提升大批量数据集处理性能，并新增每周自动增量同步 D 站最新元数据、刷新综合热门榜和新兴榜、固化榜单快照与运行状态的能力。最终系统应能在可观测、可恢复、可验收的流水线中完成“增量抓取 -> 增量入库 -> 聚合刷新 -> 榜单更新 -> 导出校验 -> 状态记录”。

## 范围与约束
- In scope:
  - 建立真实数据规模下的性能基线、耗时分解和数据库查询计划记录。
  - 优化 JSONL 导入、tag/post_tag 写入、tag 计数、月度聚合、榜单构建等大数据处理链路。
  - 新增每周增量更新流水线，默认只同步元数据，不下载图片。
  - 将每次自动运行写入 `sync_job`、`sync_shard`、`job_log` 与 `ranking_snapshot` / `ranking_snapshot_item`。
  - 为增量更新和性能优化补充测试、文档、运维命令与验收证据。
- Out of scope:
  - 未经用户明确授权，不执行全站图片下载或榜单样本图片下载。
  - 本阶段不引入 ClickHouse、Spark、Airflow、Kafka 或多机集群，除非基线证明 PostgreSQL 单机优化无法满足目标。
  - 不改变当前综合榜和新兴榜的业务口径，除非后续专门确认评分公式调整。
- Constraints:
  - PostgreSQL 仍是唯一正式数据库；JSONL 保留为原始可复核输入。
  - Danbooru API 访问必须遵守限速与可恢复策略，自动任务失败后应能从检查点或 manifest 恢复。
  - 增量同步应以时间窗口和已有最大 `post.created_at` / `post.id` 为边界，避免重复全量重跑。
  - 所有优化必须有基线前后对比，不以“感觉变快”作为验收依据。

## 执行阶段

### Phase 1: 性能基线与瓶颈定位
- Purpose: 明确当前大批量处理慢在哪里，建立后续优化对比基准。
- Outputs:
  - 性能基线报告，至少包含 24 个月正式数据或可代表性大样本的导入、tag 计数、聚合、榜单构建耗时。
  - 数据规模快照：`post`、`tag`、`post_tag`、`character_monthly_stats`、`ranking_snapshot_item` 行数。
  - 慢 SQL 和 `EXPLAIN (ANALYZE, BUFFERS)` 记录，覆盖导入后计数、月度聚合、综合榜、新兴榜关键查询。
- Completion criteria:
  - 能把总耗时拆分到具体脚本或 SQL 步骤，明确 Top 3 瓶颈。
  - 产出可重复执行的基线命令和记录格式。
- Validation:
  - 在同一数据库、同一输入数据、同一参数下重复执行至少一次核心链路，耗时偏差可解释。
  - 基线报告中每个耗时都有命令、数据规模和输出位置。
- Evidence:
  - 更新 `docs/验收报告.md` 或新增基线章节。
  - 保留基线命令与结果摘要。

### Phase 2: 导入与存储层优化
- Purpose: 降低 JSONL 到 PostgreSQL 的写入时间和重复处理成本。
- Outputs:
  - 优化 `scripts/import_jsonl_fast.py` 或新增批量导入路径，优先使用批量 upsert、临时表、COPY/staging table、批量 tag 解析和批量 `post_tag` 写入。
  - 增量入库策略：同一 post 重跑时能更新元数据并避免不必要的全量删除重插。
  - 必要索引或约束补充建议，覆盖 `post.created_at`、`post_tag(tag_id, post_id)`、`tag(category, name)`、榜单查询热点。
- Completion criteria:
  - 导入链路在代表性数据集上达到明确改善目标：建议 `post_tag` 级别百万级写入较当前基线至少快 2 倍，或 24 个月全量导入总耗时降低 40% 以上。
  - 优化后数据正确性与当前口径一致。
- Validation:
  - `python -m pytest -q`
  - `python -m compileall backend cli scripts tests`
  - 对同一 JSONL 导入前后比对 `post`、`tag`、`post_tag` 行数与 Top N tag 计数。
- Evidence:
  - 性能前后对比表。
  - 数据一致性 SQL 或校验脚本输出。

### Phase 3: 聚合与榜单构建优化
- Purpose: 将榜单刷新从全表重扫转为更适合周期运行的聚合与快照流程。
- Outputs:
  - 月度聚合支持按受影响月份增量刷新，避免每周任务都全量删除重建。
  - 综合榜和新兴榜优先读取 `character_monthly_stats` / `character_monthly_copyright`，减少直接扫描全量 `post_tag`。
  - 首现信息、版权共现、recent count 等批量查询保持集合化，避免 N+1 查询。
- Completion criteria:
  - 24 个月数据上综合榜和新兴榜构建耗时达到可运营水平：建议单榜 Top 200 在 60 秒内完成；若数据规模更大，以 Phase 1 基线为准至少降低 50%。
  - 增量聚合只重算新增或变更 post 覆盖的月份，历史月份不被无必要重建。
- Validation:
  - 综合榜 JSON/CSV 通过 `scripts/validate_character_export.py`。
  - 新兴榜 JSON/CSV 通过 `scripts/validate_emerging_export.py`。
  - 优化前后 Top 200 排名差异可解释；无业务口径变化时应保持一致。
- Evidence:
  - 榜单构建耗时对比。
  - 最新 `ranking_snapshot` 与 `ranking_snapshot_item` 行数。

### Phase 4: 每周增量更新流水线
- Purpose: 实现自动每周获取 D 站最新元数据并刷新榜单。
- Outputs:
  - 新增一个可直接调度的脚本或 CLI，例如 `scripts/weekly_update_rankings.py`。
  - 流水线步骤包括：计算增量窗口、调用 `sync_recent_posts.py` 或等效同步逻辑、导入新增 JSONL、刷新受影响聚合、构建综合榜、构建新兴榜、校验导出、写入任务状态。
  - 支持 `--dry-run`、`--start-date`、`--end-date`、`--resume`、`--output-root`、`--database-url`、`--job-key` 等参数。
  - 提供 Windows Task Scheduler / cron / Docker Compose 调度示例，默认每 7 天运行一次。
- Completion criteria:
  - 在测试窗口内可以从上次成功水位线继续，只处理新增时间段。
  - 任一步失败时任务状态为 failed，并保留错误日志；修复后可 resume 或按同一窗口重跑。
  - 成功运行后生成新的 `ranking_snapshot`，导出文件更新时间和数据库快照一致。
- Validation:
  - 使用小窗口或 mock 数据跑通端到端增量更新。
  - 重复运行同一窗口不会造成重复 post、重复 post_tag 或异常膨胀的 tag 计数。
  - `sync_job.completed_shards`、`failed_shards`、`job_log` 与 manifest 一致。
- Evidence:
  - 一次成功 dry-run 记录。
  - 一次真实小窗口增量运行记录。
  - 调度配置文档。

### Phase 5: 可观测性、恢复与运维交付
- Purpose: 让自动更新可长期运行、可排障、可恢复。
- Outputs:
  - 运行摘要报告：每次运行记录抓取条数、导入条数、重复数、错误数、耗时、输出榜单路径、snapshot id。
  - 失败重试策略：分片失败补跑、导入失败回滚、榜单构建失败后的上一次快照保留策略。
  - 运维文档：如何手动触发、如何检查最新榜单、如何回滚到上一期榜单、如何补跑指定日期窗口。
- Completion criteria:
  - 普通运维人员可按文档定位最近一次任务成功/失败原因。
  - 榜单更新失败不会破坏上一版可用榜单。
- Validation:
  - 人为制造一个小窗口失败场景，确认日志、状态、重跑路径可用。
  - 检查最近两次 snapshot 能共存，并能按 `ranking_type/generated_at` 查询。
- Evidence:
  - 更新 `docs/迁移与执行指南.md` 或新增运维章节。
  - 失败恢复演练记录。
