## 目标
把 Muse DataLoad 从“适合中小规模验证的 SQLite + 脚本系统”升级成一套可长期承载近两年 Danbooru 全站元数据采集、近期/新兴热门角色分析、榜单驱动定向样本下载的正式系统。最终结果必须满足三点：一是官方榜单链路在大数据量下稳定可运行；二是年龄过滤后的近期热门角色榜单与原系统设计保持一致；三是后续可以平滑迁移到更适合海量分析的数据库与任务编排架构。

## 范围与约束
- In scope:
  - 修复官方 `build_character_list.py` / `build_emerging_character_list.py` 在全量 24 个月数据上的可运行性。
  - 固化“近两年 JSONL -> 正式数据库 -> 官方榜单”的可重复链路。
  - 设计并落地系统优化改造方案，覆盖数据库、任务编排、聚合层、下载层分离。
  - 产出正式任务书、稳定化脚本和必要测试。
- Out of scope:
  - 未经用户明确指令，不执行任何图片下载。
  - 本轮不实现 ClickHouse、队列平台或完整多机集群，只做正式方案与必要前置改造。
  - 不把临时 JSONL 直出榜单当作最终正式榜单口径。
- Constraints:
  - 必须保持与原系统产品定义一致，尤其是“角色年龄按天过滤”规则。
  - 现有 canonical 文档区必须继续作为主治理区，避免再创建重复计划。
  - 远端 `yorha-lab` 已积累大量运行产物，后续修改必须优先兼容现状并可恢复。
  - 对大库兼容性的修复必须进入正式代码，而不是只靠手工临时命令。

- Active assumptions:
  - 当前远端 `muse_dataload_24m.db` 中的原始帖子与标签关系已经足够支撑官方综合榜重建。

- Locked decisions:
  - 不再把临时 JSONL 直出榜单作为最终正式榜单。
  - 正式榜单必须优先走原系统的官方脚本与数据库链路。
  - SQLite 只用于当前修复与验证，正式中长期主分析库应迁移到 PostgreSQL。
  - 保留 JSONL 作为原始真源，不让数据库成为唯一真源。
  - 下载层继续禁止全站图片下载，只接受榜单角色驱动。
- Open questions:
  - 官方综合榜产品上是否仍保留“综合热门榜”和“年龄过滤近期榜”双榜并存，还是后续要在命名与前端交互上进一步合并说明。
  - PostgreSQL 迁移时是否同步引入任务状态机表，还是分两个里程碑实施。
